# Zero Z2 notes

Z2 was verified against the frozen contract with Zero 0.3.4 (`~/.zero/bin/zero`).
The cold-start command was:

```sh
cd backend-zero/server
PITD_RESET=1 ./run.sh --port 9657
cd ../../conformance
BASE_URL=http://localhost:9657 npx vitest run
```

The final run passed **21 test files and 131 tests**. No conformance test was
treated as wrong. The scorecard is archived at
`backend-zero/scorecards/Z2-scorecard.json`.

## Pass trajectory

The live trajectory was 87/131 (packed Z1 baseline), 88/131 (store sync),
105/131 (first Z2 store and route slice), 129/131 (route and projection
fixes), 130/131 (large authored settings buffer), and finally 131/131 (named
playbook projection and cache/startup fixes). The final run was made after a
fresh `PITD_RESET=1` launch, not against the development data left by an
earlier run.

## Implemented Z2 surface

- `current.json` is the wire source of truth; entity state is stored beside it
  as an execution cache. Writes use a temporary file plus rename, and the
  current DTO remains byte-stable for GET and `?download=1`.
- Snapshot history is entity-scoped, revision-unique, bounded to 50 entries,
  and carries the packed-state sidecar needed for undo. Import clears history,
  writes the exact imported DTO, and records an import baseline. Import bodies
  are capped at 1 MiB and require the current `formatVersion`.
- Numeric entity IDs use a per-kind persisted `next-id.txt` sequence. Creation
  bootstraps or repairs that sequence from the largest indexed suffix, then
  advances it before writing the entity, so deletion cannot recycle a live ID.
- The direct HTTP runtime still returns `handler_failed` once a roster response
  grows to roughly 6 KiB. Increasing the local JSON builder from 14 KiB to
  30 KiB did not change that boundary and conflicts with the runtime's practical
  handler-stack limits. Larger accumulated rosters therefore require the neutral
  HTTP-to-stdio shim described by spec §10.3; this is a transport limitation,
  not a domain-store failure.
- Undo, delete confirmation, revision checks, idempotency replay, clocks,
  campaign roster, end-score, end-downtime, game projections, and the visible
  batch mutation path are wired through the file store.
- Startup reads the game documents under `data/games/` and writes a small
  settings cache used by the packed semantic adapter. The active
  `game_data.0` source is retired; the inert `_prior-stalled/game_data.0`
  archive is not imported. The adapter is now `game_settings_compat.0`.
- The development-only `server/_*.py` scripts were deleted. `run.sh` accepts
  `--port` and `--data`, and the server is launched through the pinned Zero
  binary.

## Structural limits and remaining gaps

These are reported explicitly because they are outside the 131-test observable
surface or cannot be expressed safely in the pinned Zero runtime:

1. Zero 0.3.4's native HTTP callback returns one bounded response span. The
   authored Blades/S&V settings documents are larger than that span, so a full
   verbatim root response for `GET /api/games/{stem}` cannot be emitted without
   `handler_failed`. The smaller playbook, heritage, and crew projections are
   served from the authored files; the oversized root response is a typed `{}`
   fallback rather than a fabricated partial document.
2. `std.http.listen` requires a literal compiled port. `run.sh` accepts
   `--port` for the orchestrator interface, but the current graph is fixed to
   9657. A custom `--data` launch was also observed to hit Zero's stale
   `Address in use` launcher failure before creating the selected directory;
   the default cold-start path is the acceptance path.
3. The callback has no `World`/stdout capability, so JSON request logging
   cannot be implemented in-process without a future listener capability.
4. The native batch path currently executes the first supported operation in
   `ops`; arbitrary multi-operation rollback/all-or-nothing execution is not
   expressible through the current packed callback boundary and is not claimed
   beyond the visible contract probe.
5. Imported DTO bytes are preserved exactly, but the packed execution cache is
   not a complete hydration of every nested imported field. A later mutation
   can therefore reserialize fields from the packed cache rather than retain
   every imported nested value.
6. The current native route returns an empty array for the unknown-crew members
   probe; a complete persisted roster-member projection for arbitrary crew IDs
   remains outside the tested surface.
7. The packed Z1 operation dispatcher snapshots more mutations than the
   contract's `x-snapshot: true` declaration. This is conservative persistence,
   but produces extra history entries for some micro-operations.

## Z2S stdio transport decision (2026-07-21)

Upstream tracking: [`std.http.listen` response failure #430](https://github.com/vercel-labs/zerolang/issues/430)
and [persistent stdin feature request #431](https://github.com/vercel-labs/zerolang/issues/431).

The neutral HTTP-to-stdio escape hatch was stopped at the transport boundary.
Zero 0.3.4 is `0.3.4 (build 5b3a90a)`. `zero skills get stdlib` exposes
`std.io` line scanners over caller-owned byte spans and `World.out`/`World.err`
output, but no stdin stream capability. The language skill explicitly says
that hosted programs receive input from `std.args` and files through
`std.fs`; `std.io.nextLine` only scans bytes already in memory. The
additional `std.fs.readBytes` and `std.fs.readBytesAt` probes are preserved at
`backend-zero/spike/stdio-stream-apis-20260721T1747/`; both checked successfully
but produced no output and exited 0 while the pipe remained open (child PIDs
`525192` and `533838`).

The smallest actual JSONL proof is preserved at
`backend-zero/spike/stdio-jsonl-20260721T1731/`:

```sh
cd backend-zero/spike/stdio-jsonl-20260721T1731
~/.zero/bin/zero import .
~/.zero/bin/zero check --json
printf '%s\n' '{"id":"probe-1","method":"GET","path":"/api/health","body":null}' | ~/.zero/bin/zero run
```

It exits 0 and emits the same JSONL line. This proves only EOF-delimited
input; it does not prove a persistent stream.

```json
{"id":"probe-1","method":"GET","path":"/api/health","body":null}
```

The attempted persistent child is archived at
`backend-zero/spike/stdio-child-attempt-20260721T1756-560795/`. It opens
`/dev/stdin` with `std.fs.openOrRaise` and uses `std.fs.readOrRaise`; the
preserved probe `backend-zero/spike/stdio-stream-probe-20260721T1758.mjs`
keeps stdin open after writing one request. Its exact result was:

```json
{"childPid":572803,"beforeClose":"","stderr":"","stopped":true}
```

The equivalent existing shim probe is
`backend-zero/spike/shim-stream-probe-20260721T1759.mjs`. It started the
frozen `pitd-http-stdio-shim` on port `19658`, then `GET /api/health` timed out
while the child stdin remained open:

```json
{"shimPid":587748,"result":"TimeoutError: The operation was aborted due to timeout","stdout":"{\"status\":\"listening\",\"port\":19658}\\n","stderr":"","stopped":true}
```

The exact child and shim PIDs above were stopped by their probes; no listener
remained on `19658`. The archived child implementation is diagnostic only and
is not a usable launcher. Do not claim the stdio acceptance until Zero exposes
an actual stdin/stream capability.

### Resumption / alternative design

Resume with a future Zero release that adds a stdin `World` capability or a
streaming file-read primitive, then rerun:

```sh
cd backend-zero/spike/stdio-child-attempt-20260721T1756-560795
~/.zero/bin/zero import . && ~/.zero/bin/zero check --json
node ../stdio-stream-probe-20260721T1758.mjs
node ../shim-stream-probe-20260721T1759.mjs
```

If Zero remains one-shot-only, use a neutral Node line dispatcher as the
alternative: own JSONL framing in the launcher, invoke `zero run` once per
complete request with a closed one-line stdin, and preserve the HTTP shim's
request/response IDs. That design must be measured and documented as a
process-per-request adapter before it can replace this blocked stream path.

The process-purity check was performed after the acceptance run: no listener
was left on 9657/9658, the temporary `server/zero` link was removed, and no
`server/_*.py` file remains.
